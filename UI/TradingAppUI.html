ℹHow Outbox Works

Transactional Outbox Pattern

  • Order create writes BOTH Order and OutboxMessage in one DB transaction.
  • ScheduledOutboxMessageProcessor polls for ProcessedAt = null rows.
  • Publishes the payload to CREATE_ORDER_QUEUE Service Bus queue.
  • Sets ProcessedAt on success. Increments RetryCount on failure.
  • Guarantees no message is published without persisting state first.

⚠Manual Chaos Scenarios
Require external action (kill services, pause queues)

DB Outage During Create

  • Stop SQL Server → POST /api/order.
  • Expect 500 with NO orphaned rows.
  • Verify Orders & OutboxMessages counts unchanged (transaction rollback).

Service Bus Down

  • Disable the queue in Azure portal → submit orders.
  • Orders persist; OutboxMessages remain unprocessed; RetryCount climbs.
  • Re-enable → dispatcher catches up; rows flip processed.

Consumer Crash Mid-Flow

  • Kill OrderExecutionProvider during a burst.
  • Service Bus redelivers; atomic WHERE IsProcessed=0 prevents duplicate updates.
  • Restart consumer — orders reach ACK/FILLED with no duplicates.

Poison Message → DLQ

  • Force consumer to always throw on a specific ClientOrderId.
  • After max delivery count, Service Bus moves it to DLQ.
  • DeadLetterQueueProcessor writes to DeadLetterLogs table.
  • Triage in the Dead Letter tab; resolve with operator notes.

Multi-Consumer Race

  • Run the function host on two machines pointed at same Service Bus.
  • Both compete for messages (queue) but receive copies (topic subscriptions).
  • Atomic UPDATE guarantees only one wins for any given order.
  • Loser logs: "already processed by another instance".

Subscriber Isolation

  • Kill ONE topic subscriber (e.g., AuditLogProcessor).
  • Other subscribers (Risk, Notification) keep receiving events.
  • Audit subscription queue accumulates messages.
  • Restart subscriber — backlog drains. Pub/sub decoupling proven.

Queue Pattern (Commands)

  • CREATE_ORDER_QUEUE = command queue
  • Competing consumers — message goes to one recipient
  • Load-balanced order execution

Topic Pattern (Events)

  • order_events_topic = event broadcast
  • 3 subscriptions: each gets its own copy
  • Independent scaling, failure isolation

Reliability Guarantees

  • Outbox: no message without DB commit
  • Idempotency: atomic UPDATE WHERE IsProcessed=0
  • DLQ: poison messages quarantined for triage
