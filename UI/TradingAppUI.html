What is the Outbox?

  • Each order writes a message into the OutboxMessages table in the same DB transaction.
  • The ScheduledOutboxMessageProcessor Function picks up unprocessed rows and publishes them to Service Bus.
  • Once dispatched, IsProcessed=true and ProcessedAt is set.
  • If publishing fails, RetryCount increments.

📚 Manual Failure Scenarios (No Automation Possible)

DB Down During Create

  • Stop SQL Server / Azurite → POST /api/order → expect 500 with no orphaned outbox rows.
  • Verify Orders count and OutboxMessages count both unchanged (transaction rollback).

Service Bus Down

  • Pause the order-events queue in Azurite/Azure → create orders.
  • Orders persist; OutboxMessages remain unprocessed; RetryCount climbs.
  • Resume queue → ScheduledOutboxMessageProcessor catches up; rows flip to processed.

Consumer Crash Mid-Processing

  • Kill OrderExecutionProvider while burst-loading.
  • Service Bus redelivers messages; idempotency via ClientOrderId + IsProcessed prevents duplicate updates.
  • Restart the function — orders reach ACKNOWLEDGED/FILLED without duplicates.

Dead-Letter Path

  • Force a poison message (e.g., consumer always throws) → Service Bus moves it to DLQ after max delivery count.
  • DeadLetterQueueProcessor picks it up and writes to DeadLetterLogs.
  • Triage via the Dead Letter tab; resolve with notes.

FIFO / Ordering

  • Burst create with 0ms delay → check that consumer logs process them roughly in submission order (Service Bus is not strictly FIFO unless sessions used).
  • Use View All Orders sorted by CreatedAt to compare against UpdatedAt ordering.
